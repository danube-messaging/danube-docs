import { Tabs, TabItem } from '@astrojs/starlight/components';

The Schema Registry provides type safety, validation, and schema evolution for your messages. Register schemas once, reference them from producers, and let the broker enforce validation.

## Workflow

1. Register a schema with the Schema Registry
2. Link the producer to the schema subject
3. Send validated messages (broker enforces schema matching)
4. Consumer fetches the schema and deserializes

## Registering Schemas

### JSON Schema

<Tabs>
<TabItem label="Rust">

```rust title="Register JSON Schema"
use danube_client::SchemaType;

let json_schema = r#"{
    "type": "object",
    "properties": {
        "user_id": {"type": "string"},
        "event": {"type": "string"},
        "timestamp": {"type": "integer"}
    },
    "required": ["user_id", "event", "timestamp"]
}"#;

let schema_id = schema_client
    .register_schema("user-events")
    .with_type(SchemaType::JsonSchema)
    .with_schema_data(json_schema.as_bytes())
    .execute()
    .await?;
```

</TabItem>
<TabItem label="Go">

```go title="Register JSON Schema"

schemaID, err := client.Schema().RegisterSchema("user-events").
    WithType(danube.SchemaTypeJSONSchema).
    WithSchemaData([]byte(jsonSchema)).
    Execute(ctx)
```

</TabItem>
<TabItem label="Java">

```java title="Register JSON Schema"

        {
          "type": "object",
```

## Compatibility Checking

Check if a new schema version is compatible before registering:

<Tabs>
<TabItem label="Rust">

```rust title="Check compatibility"
let result = schema_client
    .check_compatibility(
        "user-events",
        new_schema.as_bytes(),
        SchemaType::JsonSchema,
        None,  // Use subject's default mode
    )
    .await?;

if result.is_compatible {
    println!("Safe to register!");
} else {
    eprintln!("Incompatible: {:?}", result.errors);
}
```

</TabItem>
<TabItem label="Java">

```java title="Check compatibility"
} else {
    System.out.println("Incompatible: " + result.errors());
```

</TabItem>
<TabItem label="Python">

```python title="Check compatibility"
    "user-events",
    new_schema.encode(),

    print("Safe to register!")
else:
    print(f"Incompatible: {errors}")
```

### Compatibility Modes

Compatibility modes control schema evolution at the subject level. This is an **admin-only** operation:

| Mode | Description | Use Case |
|------|-------------|----------|
| `Backward` | New schema reads old data | Consumers upgrade first (default) |
| `Forward` | Old schema reads new data | Producers upgrade first |
| `Full` | Both backward + forward | Critical schemas |
| `None` | No validation | Development only |

```bash title="Set compatibility mode (admin only)"
danube-admin schema set-compatibility \
  --subject user-events \
  --mode BACKWARD
```

## Producer with Schema

The most common pattern: link a producer to a schema subject and send typed messages.

<Tabs>
<TabItem label="Rust">

```rust title="Schema-validated producer"
// 1. Register schema
schema_client
    .register_schema("user-events")
    .with_type(SchemaType::JsonSchema)
    .with_schema_data(schema.as_bytes())
    .execute()
    .await?;

// 2. Create producer with schema reference
let mut producer = client
    .new_producer()
    .with_topic("/default/user-events")
    .with_name("event-producer")
    .with_schema_subject("user-events")  // Uses latest version
    .build()?;

producer.create().await?;

// 3. Send typed messages
let event = serde_json::to_vec(&UserEvent {
    user_id: "user-123".into(),
    event: "login".into(),
    timestamp: 1234567890,
})?;

producer.send(event, None).await?;
```

</TabItem>
<TabItem label="Go">

```go title="Schema-validated producer"
// 1. Register schema
client.Schema().RegisterSchema("user-events").
    WithType(danube.SchemaTypeJSONSchema).
    WithSchemaData([]byte(jsonSchema)).
    Execute(ctx)

// 2. Create producer with schema reference
producer, _ := client.NewProducer().
    WithTopic("/default/user-events").
    WithName("event-producer").
    WithSchemaSubject("user-events").
    Build()

producer.Create(ctx)

// 3. Send typed messages
event := map[string]interface{}{
    "user_id":   "user-123",
    "event":     "login",
    "timestamp": 1234567890,
}
jsonBytes, _ := json.Marshal(event)
producer.Send(ctx, jsonBytes, nil)
```

</TabItem>
<TabItem label="Java">

```java title="Schema-validated producer"
// 1. Register schema
schemaClient.registerSchema(
        schemaClient.newRegistration()
                .withSubject("user-events")
                .withSchemaType(SchemaType.JSON_SCHEMA)
                .withSchemaDefinition(schema.getBytes()));

// 2. Create producer with schema reference
Producer producer = client.newProducer()
        .withTopic("/default/user-events")
        .withName("event-producer")
        .withSchemaLatest("user-events")
        .build();

producer.create();

// 3. Send typed messages
String event = "{\"user_id\": \"user-123\", \"event\": \"login\", \"timestamp\": 1234567890}";
producer.send(event.getBytes(), Map.of());
```

</TabItem>
<TabItem label="Python">

```python title="Schema-validated producer"

    schema_client.register_schema("user-events")
    .with_type(SchemaType.JSON_SCHEMA)
    .with_schema_data(json_schema.encode())
    .execute()
)

    client.new_producer()
    .with_topic("/default/user-events")
    .with_name("event-producer")
    .with_schema_subject("user-events")  # Uses latest version
    .build()
)

await producer.create()

# 3. Send typed messages
event = json.dumps({"user_id": "user-123", "event": "login", "timestamp": 1234567890})
await producer.send(event.encode())
```

### Version Selection Strategies

The examples above use `with_schema_subject`, which always resolves to the **latest** version. You can also pin to a specific version or set a minimum:

| Strategy | Method | Behavior |
|----------|--------|----------|
| **Latest** (default) | `with_schema_subject("subj")` | Always uses the newest registered version |
| **Pinned** | `with_schema_version("subj", 2)` | Locked to version 2, even if v3+ exists |
| **Minimum** | `with_schema_min_version("subj", 2)` | Uses v2 or any newer compatible version |

<Tabs>
<TabItem label="Rust">

```rust title="Version strategies"
// Pinned to version 2
.with_schema_version("user-events", 2)

// Minimum version 2 (uses latest >= 2)
.with_schema_min_version("user-events", 2)
```

</TabItem>
<TabItem label="Go">

```go title="Version strategies"
// Pinned to version 2
.WithSchemaVersion("user-events", 2)

// Minimum version 2 (uses latest >= 2)
.WithSchemaMinVersion("user-events", 2)
```

</TabItem>
<TabItem label="Java">

```java title="Version strategies"
// Pinned to version 2
.withSchemaPinnedVersion("user-events", 2)

// Minimum version — not yet supported, use withSchemaLatest
.withSchemaLatest("user-events")
```

</TabItem>
<TabItem label="Python">

```python title="Version strategies"
# Pinned to version 2
.with_schema_version("user-events", 2)

# Minimum version 2 (uses latest >= 2)
.with_schema_min_version("user-events", 2)
```

</TabItem>
</Tabs>

**When to use each:**

- **Latest**: most common, automatic upgrades when new versions are registered
- **Pinned**: legacy apps that haven't upgraded, testing specific versions, gradual rollouts
- **Minimum**: require a minimum feature set but allow automatic upgrades to compatible versions

## Consumer with Schema

Consume messages and deserialize using the schema:

<Tabs>
<TabItem label="Rust">

```rust title="Schema-validated consumer"
let mut stream = consumer.receive().await?;
while let Some(message) = stream.recv().await {
```

```go title="Schema-validated consumer"
        continue

}
```

## Broker Validation Policies

The broker validates schema fields based on the topic's validation policy:

| Policy | Behavior |
|--------|----------|
| **None** | No validation; schema fields optional |
| **Warn** | Logs warning if `schema_id` missing or invalid; allows message |
| **Enforce** | Rejects message if `schema_id` missing, invalid, or doesn't match |

Set via admin CLI:

```bash title="Set validation policy"
danube-admin topics set-validation-policy \
  --topic /default/user-events \
  --policy enforce
```
