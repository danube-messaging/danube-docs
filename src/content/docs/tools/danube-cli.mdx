---
title: Danube CLI
description: Command-line tool for producing, consuming, and testing messages with Danube.
---

`danube-cli` is a standalone binary for interacting with Danube at the **message level**. Use it to produce and consume messages, register schemas, and test your deployment from the terminal.

:::note
`danube-cli` operates on the **data plane** (messages, schemas). For cluster administration (brokers, namespaces, RBAC), use [`danube-admin`](/admin/overview/).
:::

## Installation

```bash title="Linux"
curl -L -o danube-cli https://github.com/danube-messaging/danube/releases/latest/download/danube-cli-linux
chmod +x danube-cli
sudo mv danube-cli /usr/local/bin/
```

```bash title="macOS (Apple Silicon)"
curl -L -o danube-cli https://github.com/danube-messaging/danube/releases/latest/download/danube-cli-macos
chmod +x danube-cli
sudo mv danube-cli /usr/local/bin/
```

```bash title="Verify"
danube-cli --version
```

## Quick Start

**Terminal 1** — produce 5 messages:

```bash title="Produce"
danube-cli produce \
  -s http://localhost:6650 \
  -t /default/demo \
  -m "Hello from Danube CLI!" \
  --count 5
```

```text title="Expected output"
✅ Producer 'test_producer' created successfully
📤 Message 1/5 sent successfully (ID: ...)
📤 Message 2/5 sent successfully (ID: ...)
📤 Message 3/5 sent successfully (ID: ...)
📤 Message 4/5 sent successfully (ID: ...)
📤 Message 5/5 sent successfully (ID: ...)
📊 Summary:
   ✅ Success: 5
```

**Terminal 2** — consume those messages:

```bash title="Consume"
danube-cli consume \
  -s http://localhost:6650 \
  -t /default/demo \
  -m my-first-subscription
```

```text title="Expected output"
🔍 Checking for schema associated with topic...
ℹ️  Topic has no schema - consuming raw bytes
Received message: Hello from Danube CLI!
Size: 24 bytes, Total received: 24 bytes
```

## Producing Messages

### Text and JSON

```bash title="Simple messages"
# Text message
danube-cli produce -s http://localhost:6650 -t /default/events -m "Hello!"

# JSON message
danube-cli produce -s http://localhost:6650 -t /default/events \
  -m '{"user_id":"123","action":"login","timestamp":"2024-01-15T10:30:00Z"}'
```

### Binary Files

```bash title="Send files"
danube-cli produce -s http://localhost:6650 --file image.png
danube-cli produce -s http://localhost:6650 --file data.bin
```

### Multiple Messages

```bash title="Batch sending"
# 10 messages with 500ms interval
danube-cli produce -s http://localhost:6650 \
  -m "Message" --count 10 --interval 500

# Rapid fire (100ms minimum interval)
danube-cli produce -s http://localhost:6650 \
  -m "Fast" --count 1000 --interval 100
```

### Message Attributes

```bash title="Key-value metadata"
danube-cli produce -s http://localhost:6650 \
  -m "Alert!" \
  --attributes "priority:high,region:us-west,env:production"
```

### Reliable Delivery

```bash title="Guaranteed delivery"
danube-cli produce -s http://localhost:6650 \
  -m "Critical message" --reliable
```

### Partitioned Topics

```bash title="Partitioned producing"
danube-cli produce -s http://localhost:6650 \
  -t /default/events --partitions 4 \
  -m "Partitioned message"
```

### Schema-Validated Producing

```bash title="Produce with schema"
# Use latest schema version
danube-cli produce -s http://localhost:6650 \
  -t /default/orders --schema-subject orders \
  -m '{"order_id":"ord_123","amount":99.99}'

# Pin to specific version
danube-cli produce -s http://localhost:6650 \
  -t /default/orders --schema-subject orders --schema-version 2 \
  -m '{"order_id":"ord_456","amount":149.99}'

# Auto-register schema from file
danube-cli produce -s http://localhost:6650 \
  -t /default/events --schema-file event-schema.json --schema-type json_schema \
  -m '{"event":"signup","user_id":"u_789"}'
```

## Consuming Messages

### Basic Consumer

```bash title="Subscribe and receive"
danube-cli consume \
  -s http://localhost:6650 \
  -t /default/orders \
  -m order-processors
```

The consumer automatically detects and validates against the topic's schema if one is configured.

### Subscription Types

```bash title="Subscription modes"
# Shared (default): load balanced across consumers
danube-cli consume -s http://localhost:6650 \
  -t /default/events -m shared-sub --sub-type shared

# Exclusive: single consumer, ordered processing
danube-cli consume -s http://localhost:6650 \
  -t /default/orders -m exclusive-sub --sub-type exclusive

# Failover: active/standby with automatic failover
danube-cli consume -s http://localhost:6650 \
  -t /default/critical -m ha-sub --sub-type fail-over
```

### Worker Pool Pattern

```bash title="Parallel processing"
for i in {1..4}; do
  danube-cli consume \
    -s http://localhost:6650 \
    -t /default/tasks \
    -n "worker-$i" \
    -m task-workers \
    --sub-type shared &
done
```

### Fan-Out Pattern

Multiple subscriptions on the same topic for different processing pipelines:

```bash title="Independent consumers"
danube-cli consume -t /default/orders -m order-processing &
danube-cli consume -t /default/orders -m order-analytics &
danube-cli consume -t /default/orders -m order-notifications &
```

### Scripting

```bash title="Pipe output to shell scripts"
danube-cli consume \
  -s http://localhost:6650 \
  -t /default/events \
  -m event-processor | \
while IFS= read -r line; do
  if echo "$line" | grep -q "error"; then
    echo "Alert: $line"
  fi
done
```

### Consumer Output Format

The consumer output varies depending on whether the topic has a schema:

```text title="Raw text message"
Received message: Hello, World!
Size: 13 bytes, Total received: 13 bytes
```

```text title="JSON message with schema validation"
🔍 Checking for schema associated with topic...
✅ Topic has schema: orders (json_schema, version 2)
📥 Consuming with schema validation...
✅ Message validated against schema 'orders' (version 2)
Received message: {"order_id":"ord_123","amount":99.99}
Size: 42 bytes, Total received: 42 bytes
```

```text title="Binary data"
Received message: [binary data - 1024 bytes]
Size: 1024 bytes, Total received: 1024 bytes
```

Consumers automatically fetch the schema using the `schema_id` from message metadata, cache it locally, and validate incoming messages. No manual schema configuration is needed on the consumer side.

## Schema Management

`danube-cli` can register schemas, check compatibility, and list versions. Setting compatibility mode and deleting schemas requires [`danube-admin`](/admin/cli-resources/#schemas).

### Supported Schema Types

| Type | Description | File Extension |
|------|-------------|---------------|
| `json_schema` | JSON Schema (Draft 7) | `.json` |
| `avro` | Apache Avro | `.avsc` |
| `protobuf` | Protocol Buffers | `.proto` |
| `string` | Plain text, no validation | — |
| `bytes` | Raw bytes, no validation | — |

### Register

```bash title="Register a schema"
# JSON Schema (most common)
danube-cli schema register user-events \
  --schema-type json_schema \
  --file user-schema.json

# Avro schema
danube-cli schema register payment-events \
  --schema-type avro \
  --file payment.avsc

# Inline schema
danube-cli schema register simple \
  --schema-type json_schema \
  --schema '{"type":"object","properties":{"id":{"type":"string"}}}'
```

### Get Details

```bash title="Retrieve schema"
danube-cli schema get user-events
danube-cli schema get user-events --output json
```

### List Versions

```bash title="Version history"
danube-cli schema versions user-events
```

### Check Compatibility

```bash title="Validate before registration"
danube-cli schema check user-events \
  --schema-type json_schema \
  --file user-schema-v2.json
```

### Schema Evolution Workflow

```bash title="Safe schema update"
# 1. Check current schema
danube-cli schema get orders

# 2. Check compatibility of new version
danube-cli schema check orders \
  --schema-type json_schema \
  --file orders-v2.json

# 3. Register if compatible
danube-cli schema register orders \
  --schema-type json_schema \
  --file orders-v2.json

# 4. Verify
danube-cli schema versions orders
```

## Troubleshooting

### Schema Not Found

```bash title="Verify and register"
# Check if schema exists
danube-cli schema get my-subject

# If not found, register it
danube-cli schema register my-subject \
  --schema-type json_schema --file schema.json
```

### Validation Failures

If a producer receives a validation error, check that the message matches the schema's required fields and types:

```bash title="Inspect the active schema"
danube-cli schema get my-subject --output json | jq .
```

### Compatibility Errors

```bash title="Check before registering"
danube-cli schema check my-subject \
  --schema-type json_schema --file new-schema.json
```

If the check fails, review the compatibility mode with `danube-admin schemas get-compatibility my-subject` and adjust your schema accordingly.

:::tip
Every command supports `--help` with built-in examples:
```bash
danube-cli produce --help
danube-cli consume --help
danube-cli schema --help
```
:::

## Command Reference

### Producer Flags

| Flag | Short | Description | Default |
|------|-------|-------------|---------|
| `--service-addr` | `-s` | Broker URL | `http://127.0.0.1:6650` |
| `--topic` | `-t` | Topic name | `/default/test_topic` |
| `--message` | `-m` | Message content | Required |
| `--file` | `-f` | Binary file path | — |
| `--producer-name` | `-n` | Producer name | `test_producer` |
| `--schema-subject` | — | Schema subject (latest version) | — |
| `--schema-version` | — | Pin to specific version | — |
| `--schema-min-version` | — | Use minimum version or newer | — |
| `--schema-file` | — | Schema file (auto-register) | — |
| `--schema-type` | — | Schema type | — |
| `--count` | `-c` | Number of messages | `1` |
| `--interval` | `-i` | Interval between messages (ms) | `500` |
| `--partitions` | `-p` | Number of partitions | — |
| `--attributes` | `-a` | Key-value attributes (`key:val,key2:val2`) | — |
| `--reliable` | — | Enable reliable delivery | `false` |

### Consumer Flags

| Flag | Short | Description | Default |
|------|-------|-------------|---------|
| `--service-addr` | `-s` | Broker URL | `http://127.0.0.1:6650` |
| `--topic` | `-t` | Topic name | `/default/test_topic` |
| `--subscription` | `-m` | Subscription name | Required |
| `--consumer-name` | `-n` | Consumer name | `consumer_pubsub` |
| `--sub-type` | — | `shared`, `exclusive`, or `fail-over` | `shared` |

### Schema Commands

| Command | Description |
|---------|-------------|
| `schema register <subject>` | Register or version a schema |
| `schema get <subject>` | Get latest schema details |
| `schema versions <subject>` | List all versions |
| `schema check <subject>` | Check compatibility of new schema |

### Client vs Admin Operations

| Operation | `danube-cli` | `danube-admin` |
|-----------|:---:|:---:|
| Register schemas | ✓ | ✓ |
| Get/list schemas | ✓ | ✓ |
| Check compatibility | ✓ | ✓ |
| Set compatibility mode | — | ✓ |
| Delete schema versions | — | ✓ |
| Configure topic schema | — | ✓ |
| Produce/consume messages | ✓ | — |
| Manage brokers/topics/namespaces | — | ✓ |
| RBAC (roles, bindings, tokens) | — | ✓ |
